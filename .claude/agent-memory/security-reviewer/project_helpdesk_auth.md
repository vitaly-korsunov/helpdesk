Stack: Bun monorepo, client/ (React 19 + react-router) + server/ (Express 5 + better-auth 1.6.19 + Prisma/Postgres). Auth is email/password only, sign-up disabled (`disableSignUp: true`), admin user provisioned via `server/prisma/seed.ts` using `ADMIN_EMAIL`/`ADMIN_PASSWORD` env vars.

## Recurring/tracked findings (as of 2026-06-20 review)

1. **HIGH, still open**: No rate limiting on `/api/auth/sign-in/email`. better-auth's rate limiter defaults to `enabled: options.rateLimit?.enabled ?? isProduction` (see `server/node_modules/better-auth/dist/context/create-context.mjs` ~line 165), and `isProduction` checks `NODE_ENV === "production"`. This repo never sets `NODE_ENV` anywhere (not in `.env.example`, not in any script, no Dockerfile/deploy config exists yet). `server/src/auth.ts` does not pass a `rateLimit` key at all. Net effect: rate limiting is OFF in every environment this app currently runs in, including whatever passes for "prod" until someone wires up NODE_ENV or an explicit `rateLimit: { enabled: true }`. Fix: add `rateLimit: { enabled: true, window: ..., max: ... }` explicitly in auth.ts rather than relying on NODE_ENV.

2. **MEDIUM, partially addressed**: `requireRole(...roles: Role[])` middleware added to `server/src/index.ts` (was uncommitted/working-tree-only as of the 2026-06-20 review — check git status before assuming it's merged). Implementation is logically correct: calls `auth.api.getSession`, 401s if no session, 403s if `!roles.includes(session.user.role as Role)`. The `as Role` cast is safe in practice because: (a) `auth.ts` sets `role: { input: false }` on the additionalFields config, so role can never be set via better-auth's own API input parsing (sign-up/update-user) — only directly via Prisma; (b) Prisma schema enforces `role Role @default(AGENT)` as a real Postgres enum, so the column can never hold a value outside `ADMIN`/`AGENT` via normal Prisma writes; (c) session reads are NOT served from better-auth's cookie-cache/JWE path here because that cache only activates when `!isStateful` (no DB session store) — this app uses `prismaAdapter`, so `getSession` always hits Postgres fresh. So the cast doesn't currently mask a real runtime risk, but it's still a defensive-coding nit (no fallback/validation if the enum or the `input:false` config ever changes). NOT YET APPLIED to any route — `/api/tickets` is intentionally open to any authenticated user, and there's no admin-only endpoint yet (`/user` page on the client is a static placeholder with no backing API).

3. **MEDIUM, fixed correctly**: CORS origin fallback removed. `server/src/index.ts` now reads `CLIENT_URL` once into a `clientUrl` const, throws at startup if unset (same pattern as `server/src/auth.ts`), and reuses that same const for the `cors()` origin option. No duplicate env reads, no ordering issues — the throw-check happens before `cors()` middleware is registered.

## Fresh findings from 2026-06-20 follow-up review (new, not in original review)

- Email enumeration: NOT exploitable currently. `/sign-in/email` does dummy password-hash work on every not-found/no-credential-account/no-password branch before returning a uniform `INVALID_EMAIL_OR_PASSWORD` (see better-auth's own `api/routes/sign-in.mjs`) — this is upstream framework behavior, already mitigates timing+enumeration on login. `/sign-up/email` always 400s uniformly because `disableSignUp: true` short-circuits before any DB lookup. Password-reset (`/request-password-reset`) always 400s with `RESET_PASSWORD_DISABLED` because `auth.ts` never configures `emailAndPassword.sendResetPassword` — this also fails closed/uniformly, so no enumeration vector opened there either, but it does mean **forgot-password is completely non-functional** for end users right now (product gap, not security bug — flag if a "forgot password" link exists in the UI).
- Session cookies: `httpOnly: true`, `sameSite: lax`, `secure` tied to `BETTER_AUTH_URL`/detected protocol — all framework defaults, not overridden. No CSRF token on plain Express routes (`/api/tickets`) but `sameSite=lax` + CORS locked to `CLIENT_URL` covers the common cross-site POST case. better-auth's own auth.* endpoints have built-in origin-check/CSRF middleware gated on `trustedOrigins` (`auth.ts` sets this correctly to `[clientUrl]`).
- No `Account.password` exposure: `parseAccountOutput` strips `password`/tokens from any account object returned via API — verified in `db/schema.mjs`. Not custom code, framework guarantee.

## Follow-up review 2026-06-24 — axios/TanStack Query migration + /api/users re-check

- **requireRole/requireAuth location moved**: both now live in `server/src/middleware.ts` (previously inline in `index.ts` per the 2026-06-20 note above — that's now stale, middleware.ts is the current home and has its own `middleware.test.ts`).
- **/api/users and /api/tickets re-verified, no regression**: `index.ts` still wires `GET /api/users` through `requireRole(Role.ADMIN)` and both ticket routes through `requireAuth`. Matches `e2e/users-api.spec.ts` (new file, untracked at review time) which asserts 401 unauthenticated / 403 agent / 200 admin + an explicit field-allowlist check on the response shape, via Playwright's server-side `request` fixture (not a browser, so it does NOT exercise browser cookie/credentials behavior — see below).
- **LOW/MEDIUM finding — `client/src/lib/api.ts` axios instance has no `withCredentials: true`**: `export const api = axios.create({ baseURL: '/api' })`. Axios defaults `withCredentials` to `false`, unlike better-auth's client (`auth-client.ts` → `createAuthClient` → `@better-fetch/fetch`, which defaults to `credentials: "include"`). Currently masked in dev/e2e because `client/vite.config.ts` proxies `/api` → `localhost:3001`, making requests same-origin from the browser's point of view, so the session cookie rides along regardless of `withCredentials`. This is a latent prod-deployment bug, not yet exploitable: there's no `express.static`/SPA-serving code in `server/src/index.ts` and the app requires `CLIENT_URL` for CORS/`trustedOrigins`, implying client and server are meant to run on different origins in real deployments — the day that happens, every `api.*()` call (tickets list/create, users list) will silently 401 because the cookie won't be attached. Recommend adding `withCredentials: true` to the axios instance now, while cheap, rather than waiting for a prod incident. Not a vulnerability (fails closed, not open) but worth fixing proactively.
- Confirmed no other client-side fetch path was missed: `Home.tsx` and `UsersPage.tsx` both migrated fully from raw `fetch` to the shared `api` axios instance — no leftover bare `fetch()` calls to `/api/*` that would have inconsistent credential behavior.
- Noted: a stray/duplicate memory file exists at `client/.claude/agent-memory/security-reviewer/helpdesk_auth_architecture.md` (untracked, wrong location — should be under root `.claude/agent-memory/`, not nested under `client/`). Contains an earlier architecture writeup, superseded by this file. Flag to the user if doing cleanup; not a security issue.

## Architecture notes for future reviews
- `server/src/auth.ts` and `server/src/index.ts` both independently read `process.env.CLIENT_URL` and throw if missing — this is the established pattern for required env vars in this repo. Any new required env var should follow the same "const at top of file, throw if falsy" pattern.
- Role enum lives in `server/prisma/schema.prisma` (`enum Role { ADMIN AGENT }`), imported into TS via `../generated/prisma/enums` (Prisma generates this; don't expect to find it hand-written).
- `requireAuth` and `requireRole` are both defined inline in `server/src/index.ts` — no separate middleware file yet. If this grows, watch for the two falling out of sync (e.g. someone fixes a bug in one but not the other).
- `client/src/pages/UsersPage.tsx` is currently a static placeholder (no fetch, no data) — the `/user` route is gated client-side only via `session.user.role === 'ADMIN'` in `App.tsx`. This is fine for now since there's no sensitive data rendered, but the moment this page starts fetching a user-list API, that endpoint MUST use `requireRole(Role.ADMIN)` server-side — client-side route gating alone is not authorization.
