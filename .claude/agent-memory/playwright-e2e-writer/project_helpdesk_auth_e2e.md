As of 2026-06-20, `e2e/login.spec.ts`, `e2e/logout.spec.ts`, and `e2e/session.spec.ts` together give comprehensive coverage of the authentication system (login validation, failed sign-in, submitting state, logout, session persistence, unauthenticated deep-link). `e2e/authorization.spec.ts` covers role-based access separately (non-admin blocked from `/user`) — don't duplicate that ground.

## Concrete facts confirmed by hitting the real endpoint (not guessed)
- better-auth's generic sign-in failure message, verified via `curl -X POST http://localhost:3002/api/auth/sign-in/email` against the running test server, is **exactly** `"Invalid email or password"` (code `INVALID_EMAIL_OR_PASSWORD`) — identical for wrong-password and non-existent-email cases. This confirms there's no UI-visible enumeration leak. Use `page.getByText(/invalid email or password/i)` to assert it.
- The error/field-error component (`client/src/components/ui/field.tsx` `FieldError`) renders a `<div role="alert" data-slot="field-error">` — `page.getByRole('alert')` is sufficient; no need to also match `[data-slot="field-error"]`, they're the same element.
- `Login.tsx`'s submit button text swaps to exactly `"Signing in…"` (with the real ellipsis character `…`, not three periods) while `isSubmitting` is true, and is `disabled` in that state. To assert this without flakiness, intercept `**/api/auth/sign-in/email` with `page.route` and inject an artificial delay (e.g. 300ms via `setTimeout` then `route.continue()`) so the submitting state has time to render before resolving.
- Zod schema in `Login.tsx`: empty email → `"Email is required"`; malformed email → `"Enter a valid email"`; empty password → `"Password is required"`. These are two distinct messages for email (required vs format), not one shared "invalid" message.
- To verify no network call fires for pure client-side validation failures (e.g. empty email), use `page.route('**/api/auth/sign-in/email', ...)` with a flag set inside the handler, then assert the flag is still `false` after submit — cheap and avoids relying on timing.

## Test layout decision (and why)
- Extended `e2e/login.spec.ts` in place (rather than a separate `auth-edge-cases.spec.ts`) with two `test.describe` blocks: "client-side field validation" (empty email/password, malformed email) and "failed sign-in attempts" (wrong password, non-existent email, identical-error-text check, retry-after-failure flow), plus one top-level test for the "Signing in…" submitting state. Rationale: all of these are variations on "what happens when you try to log in" — same file's existing scope, just deepened. Kept `logout.spec.ts` and `authorization.spec.ts` untouched since they're already tightly scoped.
- Added a new `e2e/session.spec.ts` for cross-cutting session concerns that don't belong to "login" or "logout" specifically: session survives a reload while authenticated (inverse of what `logout.spec.ts` checks), and an unauthenticated user hitting a deep link (`/user`) gets the login form via `App.tsx`'s `useSession()` gate rather than a crash/blank page.
- Did NOT add: a UI signup test (no signup UI exists, `disableSignUp: true` in `server/src/auth.ts`), a 429 rate-limit test (disabled outside `NODE_ENV=production`, e2e runs with rate limiting off by design — see [[project-helpdesk-auth]] memory in the security-reviewer namespace for the history here), or an unverified-email-account test (would require extending `server/prisma/seed.e2e.ts`, which is a shared fixture other specs may depend on — flagged to the user as a suggestion rather than just doing it).

## Fixed flake in e2e/ui.spec.ts ticket-count assertions (2026-06-21)
All three "ticket form" tests captured the "before" `{n} total` count via a one-shot `.textContent()` call placed right after `login()` returned. `Home.tsx`'s `tickets` state starts as `[]` and `GET /api/tickets` resolves asynchronously, so there's a real (if usually narrow) window where `{tickets.length} total` renders `"0 total"` before the fetch resolves and re-renders with the true count. Whichever test's `.textContent()` call happened to land in that window captured `"0 total"` as its baseline and then failed once the real count (legitimately non-zero, accumulating across e2e runs since there's no truncate endpoint) loaded a moment later. This is exactly the `expect(locator).toHaveText(expected)` / "Received: 10 total" failure mode — not an app bug.

Fix applied: in each of the three tests, register `const ticketsLoaded = page.waitForResponse(res => res.url().includes('/api/tickets') && res.request().method() === 'GET')` *before* calling `login()`, then `await ticketsLoaded` immediately after `login()` resolves, and only then read the baseline count text/number. This guarantees the baseline is read post-settle, not mid-fetch. Confirmed stable across 3 consecutive full `bun run test:e2e` runs (19/19 each) after the fix. General lesson for this app: any test reading `{n} total` (or anything else driven by a `useEffect` fetch with an empty-array initial state) needs to wait for that fetch's response, not just for `login()`'s "Sign out" button visibility — `login()` only proves auth succeeded, not that `Home`'s own data fetches have resolved.

## UsersPage redesign: <li> list → real shadcn <Table> (2026-06-21)
`client/src/pages/UsersPage.tsx` was redesigned from `<li>`-per-user to a real shadcn `<Table>` (`client/src/components/ui/table.tsx` — plain semantic `<table>/<thead>/<tbody>/<tr>/<th>/<td>`, so native ARIA table roles work for free, no extra `role=` attrs needed). Exactly 4 columns now: Name, Email, Role, Created — in that order. The "unverified" `Badge` that used to render next to `!emailVerified` users (mentioned in the entry above) was removed entirely; the new design has no verification-status indicator at all.

This broke `e2e/ui.spec.ts`'s `users page` describe block, which used `getByRole('listitem')` and asserted `unverified` badge non-visibility. Fixed by switching to `getByRole('row').filter({ hasText: <email> })` for row scoping (note: `getByRole('row')` also matches the header `<tr>`, but filtering by a seeded user's email naturally excludes it since that text only appears in data rows) and `getByRole('cell', { name: 'ADMIN'|'AGENT', exact: true })` for the role-badge cell. Removed the dead "unverified badge not visible" assertions (the feature is gone, not hidden — asserting absence of removed DOM tested nothing) and replaced with a real assertion: `page.getByRole('columnheader')` collected and asserted via `toHaveText(['Name', 'Email', 'Role', 'Created'])` to lock in the exact column set and order. The nav test and the error-state test in the same file needed no changes — both were already markup-agnostic (heading text + response-wait; error text only). Confirmed stable across 2 consecutive full `bun run test:e2e` runs (24/24 each) after the fix.

General lesson: when a project migrates from ad-hoc `<li>`/`<div>` markup to real semantic HTML (tables, native form elements, etc.), prefer the resulting ARIA roles (`row`/`cell`/`columnheader` for tables) over `getByText`/`filter` soup — they're more robust to styling changes and read closer to how a real user/assistive-tech would query the page.

## Edit user dialog + PATCH /api/users/:id coverage (2026-06-24)
Added "edit user" feature coverage. A 5th `<TableHead>` ("Actions", visually-hidden via `<span className="sr-only">`) was added to the Users table for a per-row icon-only edit button (`aria-label="Edit <name>"`). Fixed the pre-existing column-header assertion in `users page` describe block: `toHaveText(['Name', 'Email', 'Role', 'Created', 'Actions'])` (was missing `'Actions'`) — confirmed this was the *only* failure before starting (17 passed/1 failed baseline).

The edit dialog is a second, independently-controlled Radix `Dialog` (`open={editingUser !== null}`, no `DialogTrigger`) rendered as a sibling to the `Card`, reusing the same field *labels* ("Name"/"Email"/"Password") as the add-user dialog but different element `id`s (`#edit-name`/`#edit-email`/`#edit-password`). Since both dialogs are mounted in the same DOM tree (Radix just toggles visibility), unscoped `page.getByLabel('Name')` etc. is ambiguous once both could theoretically be relevant — but the *actual* failure mode hit in practice was different and worse:

**Real gotcha (caused a real test failure, not hypothetical):** `page.getByLabel('Name')` matches by accessible name on *any* element, not just form controls — including the row-level edit `<button aria-label="Edit <user's name>">`. Once a test created/edited a user whose name contained the substring "Name" (e.g. `Edited Name 123`), that row's button (`aria-label="Edit Edited Name 123"`) became a second match for `getByLabel('Name')` and Playwright's strict mode threw `resolved to 2 elements`. Because `helpdesk_test` is never reset between runs, this row persists forever once created — meaning the very first run that created such a name could break *every later run* of any test using unscoped `page.getByLabel('Name'|'Email'|'Password')`, including the pre-existing `add user dialog` tests. Fixed by: (1) scoping every `getByLabel` call in both the `add user dialog` and `edit user dialog` describe blocks to `dialog.getByLabel(...)` (the open `page.getByRole('dialog')` locator) instead of `page.getByLabel(...)`, and (2) as defense in depth, avoiding the substring "Name" in any test-created user's name going forward (used names like `E2E Edited User ${Date.now()}` instead of anything containing "...Name..."). General lesson for this app: never use unscoped `getByLabel` on a page that also contains `aria-label`-only icon buttons whose label text could overlap with a form field's label word — scope to a dialog/form/row locator first.

Other facts confirmed:
- `updateUserSchema` (`core/src/schemas/user.ts`) makes `password` a plain `z.string().trim().refine(v => v === "" || v.length >= 8, ...)` — i.e. the schema itself trims, so a whitespace-only password client-side input becomes `""` before validation and before the PATCH body is sent. This matches the server's `password === "" ? null : hashPassword(...)` check exactly (`server/src/index.ts` PATCH handler) — no separate test needed for "whitespace-only password", it's equivalent to "empty password" once trimmed.
- Server PATCH conflict check correctly excludes the user's own current email (`emailTaken.id !== userId`) — changing-to-same-email is not a 409. Not separately tested (out of the requested scope) but confirmed by reading the route code.
- `page.request` (as opposed to the bare `request` fixture) shares cookies with whatever `BrowserContext` the `page` belongs to — useful for creating disposable fixtures via the API *while already logged in via the UI* in the same test, without manually copying cookies. The bare `request` fixture from `{ request }` in a test's param list is a separate, cookie-isolated `APIRequestContext` — fine for tests that only ever need their own sign-in (as `users-api.spec.ts` already did), but the wrong tool when a test needs to mix `page`-based UI state with API-level setup. Used `page.request.post('/api/users', ...)` for the happy-path edit test's disposable-user setup specifically because the test was already logged in as ADMIN via `page`.
- `POST /api/auth/sign-out` (better-auth's standard sign-out endpoint) works directly via the `request` fixture to drop a session between sign-ins within the same test — used in the new `users-api.spec.ts` `createDisposableUser` helper to create a fixture as ADMIN then immediately drop that session so the caller can sign in as whichever role it actually wants to test, all within one test function without needing separate `request` fixture instances.

Test layout: extended `e2e/ui.spec.ts` with a new `edit user dialog` describe block (sibling to `add user dialog`) — 6 tests: pre-fill check (name/email populated, password empty), happy path (change name+email, blank password, table updates, then an API-level `request.post('/api/auth/sign-in/email')` proves the ORIGINAL password still authenticates), three client-validation tests (short name/malformed email/short password via the `page.route` + boolean-flag-on-PATCH pattern), one server-conflict test (edits a disposable user's email to collide with `AGENT.email`). Extended `e2e/users-api.spec.ts` with a `PATCH /api/users/:id authorization` block (401/403/200) using a `createDisposableUser` helper that signs in as ADMIN, POSTs a throwaway user, signs out, and returns the new user's `{id, email}` — keeps the seeded ADMIN/AGENT fixtures completely untouched. Confirmed stable across 2 consecutive full `e2e/ui.spec.ts` + `e2e/users-api.spec.ts` runs (27/27 both times) and one full-suite run (all 41 e2e tests, 41/41).

## Delete user (soft delete) coverage + fixing a selector collision (2026-06-24)
A "Delete" icon button (`aria-label="Delete <name>"`) was added per row next to the existing "Edit" button, disabled (`title="Admin users cannot be deleted"`) when `role === 'ADMIN'`. This broke the pre-existing edit-dialog tests: every call site used the loose `getByRole('button', { name: /Edit/ })` regex, and once a *second* button's aria-label also contains the substring "Edit" (e.g. the fixture user named "E2E Edit Target" produces `aria-label="Delete E2E Edit Target"`, which still contains "Edit" as a substring of the *name*, not the action), the regex matches both the Edit and Delete buttons in that row → strict-mode violation. Fixed all 6 call sites across `edit user dialog` tests in `e2e/ui.spec.ts` by switching to exact-string matches on the real accessible name (`{ name: 'Edit Agent', exact: true }`, `{ name: 'Edit E2E Edit Target', exact: true }`, `{ name: 'Edit Conflict Target', exact: true }`) instead of a regex — the accessible name is always exactly `` `Edit ${user.name}` `` / `` `Delete ${user.name}` ``, so `exact: true` with the literal string fully disambiguates with zero ambiguity risk going forward, unlike a substring/regex match which is only safe until some future fixture name happens to overlap. General lesson reinforced: prefer `{ name: '<literal>', exact: true }` over `{ name: /pattern/ }` for any icon-only button whose visible "text" is entirely derived from `aria-label` interpolation — regexes look safe until a second sibling button's label shares a substring.

Delete confirmation uses a shadcn/Radix `AlertDialog` (`role="alertdialog"`), a different component from the `Dialog` (`role="dialog"`) used for add/edit — `page.getByRole('alertdialog')` scopes it correctly and is never ambiguous with the regular dialog since only one of the two dialog types is ever open at a time in these tests. Confirmed empirically (not just assumed) that `alertdialog.getByRole('button', { name: 'Delete', exact: true })` resolves to exactly one element even though the dialog's title text is also "Delete user" — `AlertDialogTitle` renders as a plain div/heading-like element, not a `<button>`, so `getByRole('button', ...)` never considers it a candidate regardless of exact-match strictness. No flake or ambiguity hit in practice across 2 consecutive full-suite runs.

Server: `DELETE /api/users/:id` (`requireRole(Role.ADMIN)`, same gate as GET/POST/PATCH) soft-deletes via `User.deletedAt` (sets it, doesn't remove the row) and deletes all the target's `Session` rows in the same `$transaction`. 404 `{message:"User not found"}` if the id doesn't exist OR `deletedAt` is already set (confirmed: deleting the same disposable user twice → 204 then 404, proving the soft-delete is treated as "gone" for all read/delete purposes despite the row persisting). 403 `{message:"Admin users cannot be deleted"}` if the target's role is ADMIN, checked server-side independent of the UI's disabled button — verified by calling the seeded ADMIN's own id directly via `request.delete` (looked the id up via the existing `GET /api/users` listing pattern, same as other admin-lookup tests in this suite). A soft-deleted user's subsequent sign-in attempt hits the existing `hooks.before` check in `server/src/auth.ts` (already covered by [[project-helpdesk-auth-e2e]]'s sign-in tests for the wrong-password/unknown-email cases) and gets the same generic "Invalid email or password" — didn't re-assert the exact wording for the post-delete case, just asserted a 4xx status, since the message is intentionally indistinguishable from other auth failures by design.

Test layout: added a `delete user dialog` describe block to `e2e/ui.spec.ts` (sibling to `edit user dialog`) with 3 tests — cancel (opens alertdialog with the name interpolated into the description text, Cancel closes it, asserts via a `page.route` DELETE-flag that no request fired, row still visible), confirm (creates a disposable user via `page.request.post`, captures the `{n} total` count before using the same `waitForResponse`-before-`login` pattern as prior count-flake fixes, clicks Delete, asserts row gone + count decremented by exactly 1 + an API-level sign-in attempt with the original password now fails), and a pure-state check that the seeded Admin row's delete button `toBeDisabled()` (deliberately did NOT click it — asserting the disabled attribute is sufficient proof it can't be triggered from the UI, clicking a disabled button proves nothing extra). Added `DELETE /api/users/:id authorization` (401/403/204 matrix, mirroring the existing PATCH block's `createDisposableUser` helper shape exactly but as its own copy rather than a shared import — this suite's existing convention is one local helper per describe block, not a shared test-utils module) and a separate `DELETE /api/users/:id behavior` block in `e2e/users-api.spec.ts` with 4 focused tests (admin-can't-delete-admin 403, double-delete 404, nonexistent-id 404, soft-deleted-user-cant-sign-in) — kept these as 4 separate tests rather than consolidating further since each asserts a genuinely distinct status/message combination and combining them would have made failures harder to attribute to a specific cause.

Confirmed `response.status()` reports `204` directly (not e.g. `200`) for the empty-body success case, and Playwright's `APIResponse.ok()` is `true` for it — no surprises there, matched the task's expectation exactly on first try.

## Useful verification technique
When unsure of exact server-rendered error text instead of guessing from minified `node_modules` dist files (hard to grep, code is bundled/minified), just start the test server directly (`cd server && bun --env-file=.env.test run dev`) and `curl` the endpoint directly with `Origin: http://localhost:4173` header (required — better-auth's trustedOrigins check rejects requests without it). Confirmed this matches what the same DB user's real seeded values exercise in playwright. Remember to kill the background process afterward (`netstat -ano | grep :3002` → `taskkill //F //PID <pid>` on Windows/git-bash).
